1. Introduction to SQL

What is SQL?

üìå Concept: SQL stands for Structured Query Language. It's a special language used to talk to databases and get information from them.

Think of SQL like a librarian's language. When you want a book, you don't search the entire library yourself. Instead, you ask the librarian "Give me all books about history published after 2020." The librarian understands your request and finds exactly what you need.

Why Do We Need SQL?

üß† Logic Behind It:

Imagine a company has 1 million customer records. Without SQL, you'd have to:

  • Open every file manually
  • Read through each one
  • Write down matching records

With SQL, you write one simple command and get results instantly. SQL makes working with large amounts of data fast, easy, and accurate.

2. Creating Tables

What is a Table?

üìå Concept: A table is like an Excel spreadsheet. It has columns (fields) and rows (records). Each column stores one type of information, and each row is one complete record.

CREATE TABLE Syntax

üß† Why This Syntax?

When creating a table, we need to tell SQL:

  • What's the table name? (e.g., Students)
  • What columns do we need? (e.g., StudentID, Name, Age)
  • What type of data goes in each column? (e.g., numbers, text, dates)

üß©Syntax:

CREATE TABLE TableName ( ColumnName DataType, ColumnName DataType, ColumnName DataType );

Common Data Types:

Data Type What It Stores Example
INT Whole numbers 25, 1000, -5
VARCHAR(50) Text up to 50 characters "John Smith", "Developer"
DECIMAL(5,2) Decimal numbers 99.99, 15.50
DATE Dates 2024-01-15
BOOLEAN True or False TRUE, FALSE

Example: Creating a Students Table

üìù Example:
CREATE TABLE Students ( StudentID INT, FirstName VARCHAR(50), LastName VARCHAR(50), Age INT, GPA DECIMAL(3,2), EnrollmentDate DATE );

What this does:

  • StudentID - stores student ID numbers (integers)
  • FirstName - stores first names (text, max 50 characters)
  • LastName - stores last names (text, max 50 characters)
  • Age - stores age (whole number)
  • GPA - stores GPA like 3.85 (decimal with 2 decimal places)
  • EnrollmentDate - stores enrollment dates

3. SQL Constraints

üìå Concept: Constraints are rules we add to columns to make sure the data is correct and makes sense. They prevent bad data from entering the database.

PRIMARY KEY

üß† Problem It Solves: Each student needs a unique identifier. If two students have StudentID 1, it's impossible to tell them apart!

üß©Syntax:

CREATE TABLE Students ( StudentID INT PRIMARY KEY, FirstName VARCHAR(50) );
üìù Example:

StudentID 101 ‚Üí John
StudentID 102 ‚Üí Sarah
StudentID 103 ‚Üí Mike

Each student has ONE unique ID. No duplicates allowed.

NOT NULL

üß† Problem It Solves: Some information MUST be provided. You can't enroll a student without a name!

üß©Syntax:

CREATE TABLE Students ( StudentID INT PRIMARY KEY, FirstName VARCHAR(50) NOT NULL );
üìù Example:

‚úÖ Allowed: INSERT StudentID=101, FirstName="John"
‚ùå Rejected: INSERT StudentID=102, FirstName=NULL

UNIQUE

üß† Problem It Solves: Two students can have the same name, but they shouldn't have the same email!

üß©Syntax:

CREATE TABLE Students ( StudentID INT PRIMARY KEY, Email VARCHAR(100) UNIQUE );
üìù Example:

‚úÖ Allowed: john@email.com and sarah@email.com
‚ùå Rejected: Two students with john@email.com

CHECK

üß† Problem It Solves: Age shouldn't be negative, and GPA shouldn't be above 4.0!

üß©Syntax:

CREATE TABLE Students ( StudentID INT PRIMARY KEY, Age INT CHECK (Age >= 0 AND Age <= 120), GPA DECIMAL(3,2) CHECK (GPA >= 0 AND GPA <= 4.0) );
üìù Example:

‚úÖ Allowed: Age=20, GPA=3.85
‚ùå Rejected: Age=-5
‚ùå Rejected: GPA=5.0

FOREIGN KEY

üß† Problem It Solves: We need to connect related data. A course enrollment must reference a real student and a real course!

üß©Syntax:

CREATE TABLE Enrollments ( EnrollmentID INT PRIMARY KEY, StudentID INT, CourseID INT, FOREIGN KEY (StudentID) REFERENCES Students(StudentID), FOREIGN KEY (CourseID) REFERENCES Courses(CourseID) );
üìù Example:

Students table has: StudentID 101, 102, 103
‚úÖ Allowed: Enroll StudentID 102 in a course
‚ùå Rejected: Enroll StudentID 999 (doesn't exist)

Complete Example with All Constraints

CREATE TABLE Students ( StudentID INT PRIMARY KEY, FirstName VARCHAR(50) NOT NULL, Email VARCHAR(100) UNIQUE NOT NULL, Age INT CHECK (Age >= 18), GPA DECIMAL(3,2) CHECK (GPA >= 0 AND GPA <= 4.0) );

4. Inserting Data (INSERT INTO)

üìå Concept: After creating an empty table, we need to add data to it. The INSERT INTO command adds new rows to the table.

Basic INSERT Syntax

üß©Syntax:

INSERT INTO TableName (Column1, Column2, Column3) VALUES (Value1, Value2, Value3);
üß† How It Works:
  • Tell SQL which table you're adding to
  • List which columns you're filling
  • Provide the values in the same order

Correct Examples

üìù Example 1: Adding One Student
INSERT INTO Students (StudentID, FirstName, Email, Age, GPA) VALUES (101, 'John Smith', 'john@email.com', 20, 3.85);
üìù Example 2: Adding Multiple Students
INSERT INTO Students (StudentID, FirstName, Email, Age, GPA) VALUES (101, 'John Smith', 'john@email.com', 20, 3.85), (102, 'Sarah Jones', 'sarah@email.com', 21, 3.90), (103, 'Mike Brown', 'mike@email.com', 19, 3.75);
üìù Example 3: Adding Data in Different Order
INSERT INTO Students (FirstName, StudentID, Email, GPA, Age) VALUES ('Emma Davis', 104, 'emma@email.com', 3.92, 20);

‚úÖ This works! SQL matches the column names to the values, so order doesn't matter as long as they correspond.

Common Mistakes Beginners Make

‚ö†Ô∏è Mistake 1: Forgetting Quotes for Text
‚ùå WRONG: INSERT INTO Students (StudentID, FirstName) VALUES (101, John); ‚úÖ CORRECT: INSERT INTO Students (StudentID, FirstName) VALUES (101, 'John');

Why: Text values must be in single quotes. Without quotes, SQL thinks "John" is a column name, not a value.

‚ö†Ô∏è Mistake 2: Wrong Number of Values
‚ùå WRONG: INSERT INTO Students (StudentID, FirstName, Email) VALUES (101, 'John', 'john@email.com', 20);

Why: You listed 3 columns but provided 4 values. SQL won't know which value goes where.

‚ö†Ô∏è Mistake 3: Violating Constraints
‚ùå WRONG: INSERT INTO Students (StudentID, FirstName, Age, GPA) VALUES (101, 'John', -5, 5.0);

Why: Age can't be negative (violates CHECK), and GPA can't be 5.0 (violates CHECK). This insert will be rejected!

‚ö†Ô∏è Mistake 4: Duplicate Primary Key
‚ùå WRONG: INSERT INTO Students (StudentID, FirstName) VALUES (101, 'John'); INSERT INTO Students (StudentID, FirstName) VALUES (101, 'Sarah');

Why: StudentID 101 already exists. PRIMARY KEY must be unique. The second insert will fail.

5. Aggregate Functions

üìå Concept: Aggregate functions perform calculations on multiple rows and return a single result. They help you summarize data.

COUNT - How Many?

üß† Purpose: Counts the number of rows that match a condition.

üß©Syntax:

SELECT COUNT(*) FROM Students;
üìù Real-Life Analogy: A teacher counts attendance. "How many students are here today?" COUNT answers this.
SELECT COUNT(*) FROM Students WHERE Age > 20;

Returns: The number of students older than 20

SUM - Add Them Up

üß† Purpose: Adds up all values in a column.

üß©Syntax:

SELECT SUM(ColumnName) FROM TableName;
üìù Real-Life Analogy: A cashier adds up all purchase prices. "What's the total bill?" SUM answers this.
SELECT SUM(Price) FROM Orders;

Returns: The total value of all orders

AVG - Find the Average

üß† Purpose: Calculates the average of all values in a column.

üß©Syntax:

SELECT AVG(ColumnName) FROM TableName;
üìù Real-Life Analogy: A weather service calculates average temperature. "What's the average temperature this month?" AVG answers this.
SELECT AVG(GPA) FROM Students;

Returns: The average GPA of all students (e.g., 3.83)

MAX and MIN - Extremes

üß† Purpose: Finds the highest (MAX) or lowest (MIN) value.

üß©Syntax:

SELECT MAX(ColumnName) FROM TableName; SELECT MIN(ColumnName) FROM TableName;
üìù Real-Life Analogy: In a race, "Who was the fastest?" (MAX speed) or "Who was slowest?" (MIN speed)
SELECT MAX(GPA) FROM Students; -- Highest GPA SELECT MIN(Age) FROM Students; -- Youngest student

All Aggregate Functions Together

SELECT COUNT(*) as TotalStudents, AVG(GPA) as AvgGPA, MAX(GPA) as HighestGPA, MIN(Age) as YoungestAge, SUM(StudentID) as SumOfIDs FROM Students;

6. GROUP BY and HAVING

üìå Concept: GROUP BY groups rows with similar values together, then applies aggregate functions to each group. HAVING filters these grouped results.

Why GROUP BY is Needed

üß† Problem It Solves:

Imagine you're a teacher with students in different majors:

  • Math: John, Sarah, Emma
  • Computer Science: Mike, Lisa, Alex
  • Physics: David, Rachel

If you ask "What's the average GPA?", you get ONE number for everyone. But if you ask "What's the average GPA BY MAJOR?", you want three numbers - one for each major!

That's what GROUP BY does!

GROUP BY Syntax and Example

üß©Syntax:

SELECT ColumnToGroup, AGGREGATE_FUNCTION(ColumnToCalculate) FROM TableName GROUP BY ColumnToGroup;
üìù Example: Average GPA by Major
SELECT Major, AVG(GPA) as AverageGPA FROM Students GROUP BY Major;

Result might look like:

Major AverageGPA
Math 3.87
Computer Science 3.92
Physics 3.75
üìù Example: Count Students by Age
SELECT Age, COUNT(*) as NumberOfStudents FROM Students GROUP BY Age;

Result might look like:

Age NumberOfStudents
19 5
20 8
21 6

What is HAVING?

üß† Why We Need HAVING:

WHERE filters rows BEFORE grouping, but HAVING filters groups AFTER grouping.

Example: You want only majors with average GPA above 3.8. HAVING lets you filter grouped results!

GROUP BY with HAVING Syntax

üß©Syntax:

SELECT ColumnToGroup, AGGREGATE_FUNCTION(Column) FROM TableName GROUP BY ColumnToGroup HAVING AGGREGATE_FUNCTION(Column) > Value;
üìù Example 1: Majors with Average GPA Above 3.8
SELECT Major, AVG(GPA) as AverageGPA FROM Students GROUP BY Major HAVING AVG(GPA) > 3.8;

Result:

Major AverageGPA
Math 3.87
Computer Science 3.92

‚ùó Physics (3.75) is excluded because it doesn't meet the HAVING condition!

üìù Example 2: Age Groups with More Than 5 Students
SELECT Age, COUNT(*) as NumberOfStudents FROM Students GROUP BY Age HAVING COUNT(*) > 5;

Result:

Age NumberOfStudents
20 8

Age 19 (5 students) and Age 21 (6 students) - Wait! Age 21 has 6 students, so it WILL show.

üìù Example 3: Departments with Total Salary Bill Over $500,000
SELECT Department, SUM(Salary) as TotalSalary FROM Employees GROUP BY Department HAVING SUM(Salary) > 500000;

GROUP BY vs WHERE vs HAVING

Clause When Used Filters What? Example
WHERE Before grouping Individual rows WHERE Age > 20
GROUP BY Group rows Organizes data into groups GROUP BY Major
HAVING After grouping Groups (aggregated results) HAVING AVG(GPA) > 3.8

Complete Example: WHERE + GROUP BY + HAVING

SELECT Major, AVG(GPA) as AverageGPA, COUNT(*) as StudentCount FROM Students WHERE Age >= 19 GROUP BY Major HAVING COUNT(*) > 2;

What this does step-by-step:

  1. WHERE Age >= 19 ‚Üí Filter: Keep only students 19 years or older
  2. GROUP BY Major ‚Üí Organize: Group remaining students by major
  3. HAVING COUNT(*) > 2 ‚Üí Filter groups: Keep only majors with more than 2 students
  4. SELECT ‚Üí Show: Major name, average GPA, and count per major

7. Joins - Combining Tables

üìå Concept: Joins combine data from two or more tables based on a matching column (usually a Foreign Key).

Why Do We Need Joins?

üß† The Problem:

Information is stored in separate tables for organization:

  • Students Table: StudentID, FirstName, StudentDeptID
  • Departments Table: DeptID, DeptName, Building

To answer "What department is John in?" you need to JOIN these tables together using the matching columns.

INNER JOIN - Only Matching Records

üß† What It Does: Combines rows from both tables, but ONLY includes records that match in both tables.

SELECT Students.FirstName, Departments.DeptName FROM Students INNER JOIN Departments ON Students.DeptID = Departments.DeptID;

LEFT JOIN - Keep All Left Table Records

üß† What It Does: Keeps ALL records from the left table, and adds matching records from the right table. If no match, shows NULL/blank.

Use INNER JOIN When... Use LEFT JOIN When...

8. Subqueries - Queries Within Queries

üìå Concept: A subquery is a SQL query inside another SQL query. The inner query (subquery) runs first, and its result helps the outer query.

What is a Subquery?

üß† Real-Life Analogy:

Imagine you want to find "All employees who earn more than the average salary."

  • First: Calculate the average salary (subquery)
  • Then: Find employees earning more than that average (outer query)

A subquery lets you do both in one command!

Simple Subquery Example

üß©Syntax:

SELECT ColumnName FROM TableName WHERE ColumnName > (SELECT AVG(ColumnName) FROM TableName);
üìù Example: Find Students with Above-Average GPA
SELECT FirstName, GPA FROM Students WHERE GPA > (SELECT AVG(GPA) FROM Students);

How it works:

  1. Inner query: SELECT AVG(GPA) FROM Students ‚Üí calculates average (let's say 3.80)
  2. Outer query: Uses result ‚Üí WHERE GPA > 3.80
  3. Returns: Only students with GPA above 3.80

When to Use Subqueries

üìù Example 1: Find Students in the Same Major as John
SELECT FirstName, Major FROM Students WHERE Major = (SELECT Major FROM Students WHERE FirstName = 'John');
üìù Example 2: Find Expensive Products (Above Average Price)
SELECT ProductName, Price FROM Products WHERE Price > (SELECT AVG(Price) FROM Products);
üìù Example 3: Find Students Older Than the Youngest Student
SELECT FirstName, Age FROM Students WHERE Age > (SELECT MIN(Age) FROM Students);

Subquery vs JOIN - Which to Use?

Subquery JOIN
Better for simple calculations first Better for combining multiple table columns
Example: Find above-average items Example: Get names and department info
Easier to read for beginners Often more efficient for databases
